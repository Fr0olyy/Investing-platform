# Документация разработки

## Цели backend-а

Backend должен поддерживать 3 основных домена:

1. Пользователь и виртуальный брокерский счет.
2. Рыночные данные и торговые операции.
3. ML-подсистема для прогнозов и сценарного анализа.

## Принципы проекта

- Frontend и backend разделены.
- Бизнес-логика не должна жить в роутерах.
- Все DTO и API-контракты описываются через Pydantic.
- Внешние провайдеры данных изолируются в интеграционном слое.
- ML-часть должна быть заменяемой без слома REST-контрактов.
- Любое изменение поведения API должно отражаться в Swagger и документации.

## Слои приложения

### `api`

Задачи:

- принять HTTP-запрос;
- провалидировать вход;
- вызвать сервис;
- вернуть ответ.

### `services`

Задачи:

- реализовать бизнес-сценарии;
- работать с ORM-моделями;
- рассчитывать портфель, сделки, прогнозы;
- координировать внешние API и ML.

### `db`

Задачи:

- подключение к БД;
- ORM-модели;
- seed данных.

### `schemas`

Задачи:

- единый контракт между backend, Swagger и frontend.

### `integrations`

Задачи:

- общение с реальными внешними API;
- нормализация форматов ответов;
- защита доменного слоя от специфики XML/CSV/HTML/JSON провайдеров.

## Реальные внешние источники

Сейчас используются:

- MOEX ISS;
- Банк России;
- FRED.

Они задействованы и в backend, и в ML pipeline.

## Как добавлять новый endpoint

1. Добавить или обновить Pydantic-схемы в `app/schemas`.
2. Реализовать бизнес-логику в `app/services`.
3. Добавить маршрут в `app/api/routes`.
4. Подключить маршрут в `app/api/router.py`.
5. Добавить русские `summary` и `description`.
6. При необходимости обновить документы в `docs/`.

## Как расширять ML-часть

Текущий pipeline уже умеет:

- собирать реальные данные;
- строить датасет;
- обучать модель;
- сохранять артефакты;
- обновлять prediction cache;
- выполнять scenario inference.

Чтобы заменить baseline-модель:

1. Обновить реализацию в `app/services/ml_service.py`.
2. Сохранить прежний формат API-ответа.
3. Продолжать писать метрики и артефакты в `ml_model_metadata`.
4. Сохранять модели в `ml/artifacts`.

## Как расширять рыночную часть

Чтобы подключить нового провайдера:

1. Добавить код в `app/integrations`.
2. Не смешивать HTTP-клиент с сервисной логикой.
3. Возвращать нормализованный формат данных.
4. Подключить новый источник в `MarketService` или `MLService`.

- `DATABASE_URL`
- `MARKET_DATA_PROVIDER`
- `ENABLE_BACKGROUND_JOBS`
- `ML_AUTO_TRAIN_ON_STARTUP`
- `ML_LOOKBACK_YEARS`
- `ML_TEST_RATIO`
- `ML_MIN_DATA_POINTS`
- `ML_HORIZON_DAYS`
- `ML_ARTIFACTS_DIR`
- `ML_DATASETS_DIR`

См. шаблон:

- `.env.example`
