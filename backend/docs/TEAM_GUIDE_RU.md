# Документ для команды: полный обзор технологий, паттернов и устройства проекта

## 1. Назначение системы

Система представляет собой учебную инвестиционную платформу, где пользователь может:

- зарегистрироваться;
- получить виртуальный брокерский счет;
- смотреть реальные рыночные данные;
- торговать в режиме симулятора;
- анализировать портфель;
- использовать ML-модуль для сценарного прогноза цены.

С точки зрения архитектуры проект состоит из двух больших частей:

1. **Backend** — бизнес-логика, работа с пользователями, портфелем, сделками, рынком и API.
2. **ML subsystem** — сбор данных, обучение моделей, генерация прогнозов, сценарный анализ.

## 2. Язык и стек технологий

### Язык

Почему:

- быстрый старт;
- большой набор библиотек для web и ML;

- **python-jose** — JWT.
- **passlib + bcrypt** — хеширование паролей.
- **APScheduler** — фоновые задачи.
- **httpx** — HTTP-клиент для внешних API.

### Data / ML

- **pandas** — табличная обработка рядов.
- **scikit-learn** — обучение линейной регрессии.
- **joblib** — сериализация моделей.
- **beautifulsoup4** — парсинг HTML-таблицы ключевой ставки.

### Infra

- **Docker**
- **docker-compose**
- **PostgreSQL** в контейнере
- **SQLite** как легкий локальный fallback

## 3. Архитектурные паттерны

### Layered Architecture

Проект разделен на слои:

- `api` — HTTP слой;
- `services` — бизнес-логика;
- `db` — модели и подключение к БД;
- `schemas` — DTO / API-контракты;
- `integrations` — адаптеры внешних API.

Это нужно, чтобы:

- не смешивать веб-логику с доменной логикой;
- легче тестировать;
- проще заменять внешние источники данных;
- масштабировать проект без хаоса.

### Service Layer

Основные бизнес-сценарии вынесены в сервисы:

- `AuthService`
- `MarketService`
- `PortfolioService`
- `TradeService`
- `MLService`

Плюс:

- роутеры остаются тонкими;
- бизнес-логика живет централизованно;
- код проще поддерживать.

### DTO / Schema Pattern

Через Pydantic-схемы backend четко описывает:

- что принимает endpoint;
- что возвращает endpoint;
- как это отображается в Swagger.

### Adapter / Integration Pattern

Внешние рыночные источники не вызываются напрямую из ML или роутеров. Для этого существует:

- `app/integrations/market_data_client.py`

Это слой-адаптер, который изолирует:

- формат ответов MOEX;
- XML ЦБ;
- HTML таблицу ключевой ставки;
- CSV Brent из FRED.

### Fail-soft fallback

Если внешний источник временно недоступен:

- backend не должен падать целиком;
- система использует fallback / seed / ранее сохраненные данные;
- пользовательский сценарий остается доступен.

Это особенно важно для учебной платформы.

## 5. Как устроена ML-часть

Подробная ML-документация есть отдельно, но здесь краткая схема.

### Источники данных

- цена актива: MOEX;
- Brent: FRED;
- USD/RUB: ЦБ;
- IMOEX: MOEX;
- RGBI: MOEX;
- Key Rate: ЦБ.

### Идея модели

Для каждого тикера обучается отдельная регрессионная модель:

- вход: прошлое состояние рынка и макрофакторы;
- выход: ожидаемая цена следующего торгового дня.

### Почему отдельная модель на каждый тикер

Потому что:

- у разных компаний разная чувствительность к валюте, нефти и ставке;
- экспортеры и банки реагируют по-разному;
- единая модель для всех активов была бы слишком грубой.

### Почему Linear Regression

Как baseline она хороша тем, что:

- прозрачна;
- интерпретируема;
- быстро обучается;
- удобна для документации и защиты проекта;
- позволяет честно объяснить, как факторы влияют на цену.

## 6. Важные инженерные договоренности

### Что считается источником истины

- структура БД: ORM-модели;
- контракт API: Pydantic-схемы;
- поведение endpoint-ов: сервисы;
- внешние котировки: интеграционный слой;
- ML-артефакты: директория `ml/artifacts/`.

### Что нельзя делать

- смешивать обращения к внешним API прямо в роутерах;
- хранить бизнес-логику в Swagger-схемах;
- обучать модель “на лету” внутри пользовательского endpoint-а;
- ломать формат ответа без обновления схем и документации.
